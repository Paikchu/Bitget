# Strategy Studio 测试流程

## 目的

这份文档用于固定 Bitget 项目中 Strategy Studio 功能的标准测试流程。

目标是让 AI 或人工执行者在后续任意一次回归验证时，都可以按同一套步骤完成以下检查：

1. 在网站的“策略工作区”中填写一份自然语言策略文档。
2. 点击“生成代码”，确认系统能根据策略文档生成 Python 策略代码。
3. 检查生成代码是否与策略文档核心逻辑一致。
4. 在左侧“策略回测”中运行回测。
5. 确认左侧回测使用的是右侧最新生成的策略代码，而不是旧的固定策略。

这份文档不是设计文档，也不是实现计划。它是执行测试时的固定操作说明。

## 适用范围

本流程适用于以下场景：

- 回归验证 Strategy Studio 是否可正常使用
- 验证“策略文档 -> 生成代码 -> 左侧回测”链路是否打通
- 验证左侧回测是否真正使用右侧当前代码
- 验证生成代码的核心逻辑是否没有明显偏离策略文档

## 前置条件

执行测试前，需要满足以下条件：

1. 前端服务已启动，默认地址为 `http://127.0.0.1:5173/`
2. 后端服务已启动，默认地址为 `http://127.0.0.1:8080/`
3. `.env` 中已配置 `DEEPSEEK_API_KEY`
4. 当前 Python 虚拟环境已安装 `requirements.txt` 中的依赖，尤其是 `openai`
5. Docker 可用，因为策略回测走的是 Docker 沙箱

如果前置条件不满足，则本流程不应继续执行，应该先修复环境。

## 测试输入

执行本流程时，应使用一份新的自然语言策略文档，而不是直接使用默认内置策略。这样可以验证真正的“文档生成代码”链路。

推荐使用如下类型的策略描述：

- 包含趋势过滤
- 包含入场条件
- 包含出场条件
- 包含成交量或动量确认
- 包含明确的执行时机

推荐示例：

### 示例策略主题

`RSI 回落再突破 + EMA 趋势过滤策略`

### 示例策略必须包含的逻辑点

1. 用 `EMA20` 和 `EMA60` 过滤趋势方向
2. 用 `RSI14` 作为回落后重新转强/转弱确认
3. 用最近 `20` 根 K 线高低点作为突破条件
4. 用最近 `10` 根 K 线高低点作为保护性止损参考
5. 用最近 `10` 根 K 线平均成交量作为放量确认
6. 信号在 bar close 后确认，下一根 bar 执行

## 执行步骤

### 第一步：打开页面并确认基础入口存在

进入网站首页后，确认页面具备以下两个区域：

1. 左侧有“监控看板”
2. 右侧有“策略工作区”

同时确认以下入口存在：

- 右侧 tab：`策略文档`
- 右侧 tab：`策略代码`
- 右侧按钮：`▶ 生成代码`
- 左侧 tab：`策略回测`
- 左侧按钮：`运行回测`

### 第一步的验证目标

必须确认 Strategy Studio 页面结构完整，测试链路所需入口全部可见且可点击。

如果任一入口缺失，则本次测试应判定为失败，不进入下一步。

---

### 第二步：在“策略文档”中填写新的自然语言策略

在右侧“策略文档”编辑区中，清空默认内容或覆盖默认内容，然后填写一份新的自然语言策略文档。

要求：

1. 不能只保留默认内置策略原文
2. 文档必须明确表达趋势过滤、入场、出场、执行时机
3. 文档应尽量包含可被代码映射的定量条件

### 第二步的验证目标

必须确认编辑区中的内容已经被替换成新的测试策略，而不是仍然保留旧的默认文案。

如果文档没有被实际更新，则本次测试无效。

---

### 第三步：点击“生成代码”

在右侧“策略文档”区域点击 `▶ 生成代码`。

点击后应观察到：

1. 按钮进入“生成中...”状态
2. 请求发送到 `POST /api/strategy/generate`
3. 等待生成完成后，可以切换到“策略代码”

### 第三步的验证目标

必须确认：

1. 生成接口请求真实发出
2. 生成流程结束后右侧出现新的 Python 代码
3. 右侧代码不是空白，不是占位符，不是报错信息

如果生成接口失败、按钮无响应、代码区为空，则本次测试失败。

---

### 第四步：检查生成代码是否与策略文档一致

切换到右侧 `策略代码`，对生成结果进行一致性检查。

这里不要求逐行完全等价，但必须确认核心逻辑一致。

至少检查以下内容：

1. 是否存在 `add_indicators(df)` 和 `get_signal(df, i, params)` 两个函数
2. 是否实现了文档中提到的 `EMA20` / `EMA60`
3. 是否实现了 `RSI`
4. 是否使用了成交量条件
5. 是否实现了最近 `20` 根 K 线突破相关逻辑
6. 是否实现了最近 `10` 根 K 线止损相关逻辑

### 第四步的额外逻辑检查

如果生成代码中使用最近 N 根 K 线高低点作为突破或止损参考，还必须检查：

1. 这些 rolling 高低点不能直接和当前 bar 自身混在同一个窗口比较
2. 如果代码里出现类似 `high_20`、`low_20`、`high_10`、`low_10` 这种滚动窗口值，则读取这些窗口值时应排除当前 bar
3. 换句话说，窗口参考值应来自上一根 bar 的窗口结果，而不是当前 bar 自身

### 第四步的验证目标

必须确认生成代码与策略文档核心逻辑一致，没有明显偏题，没有明显遗漏关键条件。

允许存在实现细节差异，但不允许出现以下情况：

- 文档写了 RSI，代码完全没有 RSI
- 文档写了趋势过滤，代码没有趋势过滤
- 文档写了突破条件，代码没有突破逻辑
- 文档写了止损逻辑，代码没有止损逻辑
- 文档要求使用历史窗口，代码却把当前 bar 混进窗口比较，导致条件失真

如果出现以上任一情况，应判定为“生成代码与文档不一致”。

---

### 第五步：切换到左侧“策略回测”

点击左侧 tab `策略回测`。

进入后，确认以下内容存在：

1. 参数输入项，如天数、阈值、初始资金、杠杆、手续费
2. 按钮 `运行回测`

### 第五步的验证目标

必须确认左侧回测面板可正常进入，并具备触发回测所需的 UI。

---

### 第六步：运行回测

在左侧点击 `运行回测`。

点击后应观察到：

1. 请求发送到 `POST /api/strategy/backtest`
2. 后续轮询请求发送到 `GET /api/strategy/backtest/{job_id}`
3. 页面出现“运行中...”或类似加载状态
4. 最终页面出现回测结果，或出现明确错误信息

### 第六步的验证目标

必须确认左侧回测已经不再使用旧的固定 `/api/backtest` 逻辑，而是走新的 `/api/strategy/backtest` 链路。

如果网络请求仍然是旧的 `/api/backtest`，则说明链路未打通，应判定失败。

---

### 第七步：确认左侧回测使用的是右侧最新代码

这是本流程中最关键的一步。

验证方式如下：

1. 先让右侧生成一份合法代码，并运行一次左侧回测，确认可正常得到结果
2. 然后在右侧“策略代码”中，把代码改成明显非法的内容，例如删除 `get_signal(df, i, params)` 函数
3. 再次从左侧点击 `运行回测`

预期结果：

1. 左侧回测应立即失败或返回代码校验错误
2. 错误信息应明确指向当前代码非法，例如缺少 `get_signal`
3. 不能继续跑出正常回测结果

### 第七步的验证目标

必须确认左侧回测使用的是右侧当前最新代码，而不是缓存的旧代码，也不是项目里某个固定内置策略。

如果右侧代码改坏后，左侧回测仍然正常给出结果，则说明左侧回测没有使用当前代码，应判定失败。

## 最终验收标准

只有当以下所有条件同时满足时，本次测试才算通过：

1. 右侧“策略文档”可以输入新的自然语言策略
2. 点击“生成代码”后，能成功得到 Python 策略代码
3. 生成代码与策略文档核心逻辑一致
4. 左侧“策略回测”实际请求走的是 `/api/strategy/backtest`
5. 左侧回测可以基于合法代码成功执行
6. 左侧回测会随着右侧代码变化而变化
7. 当右侧代码非法时，左侧回测也必须失败并返回对应错误
8. 每次成功生成代码后，版本历史中会新增一条版本记录
9. 选择历史版本后，策略文档与策略代码会一起恢复到对应版本

## 失败判定规则

出现以下任一情况，都应判定本流程失败：

1. 页面入口缺失
2. 无法输入或覆盖策略文档
3. 生成接口失败
4. 生成代码为空
5. 生成代码与文档核心逻辑明显不一致
6. 左侧回测仍然调用旧的 `/api/backtest`
7. 左侧回测没有使用右侧当前代码
8. 非法代码仍能在左侧回测中跑出正常结果
9. 成功生成后没有新增版本记录
10. 载入历史版本时只恢复了代码或只恢复了文档

## 版本历史专项检查

在完成基础“文档 -> 生成代码 -> 回测”链路后，还需要额外检查版本控制功能：

1. 输入一份新的策略文档并点击 `▶ 生成代码`
2. 确认右下方或工作区下方的“版本历史”面板出现一条新的 `V{n}` 记录
3. 记录该版本号、标题、时间是否合理
4. 再修改策略文档并重新生成一次，确认版本号递增
5. 点击旧版本的 `载入`
6. 确认“策略文档”和“策略代码”同时回到旧版本内容
7. 基于旧版本内容再次点击 `▶ 生成代码`，确认又生成一个新的更高版本号

### 版本历史的验证目标

必须确认版本记录保存的是“同一时刻的策略文档 + 策略代码”快照，而不是只保存其中一侧内容。

## 建议输出格式

当 AI 执行本流程时，建议最终输出使用以下结构：

### 1. 执行结果

- 通过 / 失败

### 2. 步骤记录

- 第一步做了什么
- 第二步做了什么
- 第三步做了什么
- 每一步观察到了什么结果

### 3. 一致性检查

- 策略文档与生成代码是否一致
- 是否发现逻辑偏差
- 是否发现窗口比较错误

### 4. 回测链路检查

- 左侧是否调用 `/api/strategy/backtest`
- 左侧是否使用右侧最新代码
- 非法代码时是否能正确失败

### 5. 结论

- 是否通过
- 如果失败，失败点是什么
- 下一步建议修什么

## 文档使用方式

后续如果需要 AI 重复执行这套测试，可以直接引用本文件，并明确要求：

1. 严格按本文档步骤执行
2. 不跳步
3. 每一步都记录观察结果
4. 最终按“最终验收标准”给出通过或失败结论
